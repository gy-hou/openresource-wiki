    Agent Claude Code OpenClaw 微信 自动化

    我把整个微信社交上下文接给了 Agent¶

    这不是“做了个微信机器人”，而是把 WeFlow、ilink 和 Agent 大脑焊成了一套主动式个人管家系统。

    TL;DR¶

    我现在这套 WeChat AI Agent，不是只会回你一句“收到”的 bot，而是能读全量聊天上下文、理解关系和承诺、主动决定什么时候提醒你、什么时候闭嘴、什么时候直接执行动作的系统。

    更关键的一点是，这条路线已经不是 PPT。消息监听、消息收发和 Agent 决策主链路都已经跑通了，剩下的是把“能力”继续产品化，而不是从 0 到 1 证明它能不能做。

    先说结论：这东西为什么强¶

    大部分“聊天机器人”方案，本质上都还是单线程记事本。

    你跟 bot 说什么，它记什么；你不说，它就不知道；你不主动汇报，它就没有世界模型。这样的系统可以做提醒、做问答、做陪聊，但它理解不了你的真实生活上下文。

    而我现在这套方案，走的是完全不同的路径：

    • 输入不是 bot 对话，而是微信全量聊天
    • 理解对象不是单条消息，而是你的社交整体
    • 行动方式不是等你发号施令，而是主动触发

    这意味着 Agent 看到的，不再是一个对话框，而是一个真实的人际网络、时间流和承诺系统。

    一句话概括：

    不是给微信套一个 AI 壳，而是把你的微信社交生活变成 Agent 可计算、可推理、可执行的输入层。

    核心数据流¶

    整条主链路可以压缩成 5 步：

    1. 微信全量聊天进入系统 WeFlow 负责本地解析和监听，拿到所有私聊、群聊、历史与新增消息。
    2. 桥接层把消息路由给 Agent ilink 负责消息收发，桥接层负责协议转换、事件分发和统一入口。
    3. Agent 大脑理解上下文并做决策 当前可接 OpenClaw，后续也可以切到 Claude Code、Codex 或混合编排。
    4. 调度层决定何时触发 定时任务、模式匹配、事件触发、随机唤醒，都可以组合。
    5. 输出层把决策发回微信或写进本地系统 包括微信提醒、消息回复、本地 Dashboard、时间轴、日报周报等。

    从工程视角看，这套东西最重要的不是某个模型选型，而是数据闭环终于成立了：

    微信聊天 -> 解析 -> 决策 -> 触发 -> 输出 -> 新聊天继续回流

    一旦闭环成立，后面的功能都是在这个回路上叠能力。

    六层架构¶

    1. 数据采集层¶

    这一层解决的是“能不能拿到足够真实的数据”。

    当前已经跑通的核心是：

    • WeFlow 全量监听
    • ilink 收消息
    • 微信消息的持续接入

    后续还会补：

    • 每条消息的时间戳注入
    • 回复延迟、长度、频率等元数据提取
    • 更细粒度的会话行为特征

    如果没有这一层，后面的 Agent 再聪明，也只是空气推理。

    2. 桥接层¶

    这一层不花哨，但它决定系统能不能稳定活着。

    核心职责包括：

    • 把 WeFlow 的消息变成 Agent 能处理的统一事件
    • 把 Agent 的动作重新路由回微信发送端
    • 处理不同组件之间的协议差异

    这里可以继续接：

    • cc-connect
    • CLI-WeChat-Bridge
    • 自定义 Event Bus

    桥接层做得好，后面换模型、换调度器、换前端都不会伤筋动骨。

    3. 智能决策层¶

    这是整套系统最关键的地方。

    当前在用的是 OpenClaw，但设计上并没有把自己锁死在单一 Agent 栈里。Claude Code、Codex 都能接进来，甚至可以做分工式多 Agent。

    这一层后续要补的重点有三个：

    • 承诺提取 自动扫描“我答应你了什么”“我什么时候该交付”
    • 情绪推断 不只是看文本，还看回复延迟、长度、频率、emoji 模式
    • 上下文感知决策 判断现在该不该提醒、该怎么提醒、提醒到什么程度

    真正厉害的地方不在“能回答问题”，而在“知道什么时候别说废话，什么时候必须站出来”。

    4. 调度层¶

    如果没有调度层，Agent 永远只能被动等你叫它。

    而主动式系统的价值，恰恰来自触发逻辑：

    • 定时任务
    • 随机唤醒
    • 消息模式匹配
    • Agent 自己设置提醒队列

    这层一接上，系统的形态就从“工具”变成“管家”。

    5. 存储层¶

    要做长期价值，必须把聊天从一次性文本变成结构化资产。

    这层后续要沉淀的数据包括：

    • 承诺数据库
    • 关系图谱
    • 时间轴数据
    • 自动日记
    • 社交互动统计

    WeFlow 本地数据库已经是很强的基础，接下来要做的是把“看得见的聊天”变成“可推理的长期记忆”。

    6. 输出层¶

    最终输出不该只是一条回复消息。

    它可以是：

    • 微信提醒
    • 自动回复
    • 本地前端 Dashboard
    • 生活时间轴
    • 日报 / 周报
    • 截图式报表推送
    • 智能家居联动

    当输出层足够丰富时，Agent 才不是一个聊天框，而是你数字生活的操作台。

    这套系统最猛的 9 个能力¶

    1. 承诺追踪器¶

    你在聊天里随口答应别人的事，通常是最容易遗忘的。
    这个系统能把“社交债务”自动提取成待办，而不是等你自己想起来。

    2. 上下文感知打断¶

    不是随机 ping 你，而是判断你当下是在认真聊工作，还是已经在闲聊失控。
    这个差别，会直接决定 Agent 是不是像个有脑子的助手。

    3. 关系衰减预警¶

    重要关系并不是突然断掉的，而是缓慢降频。
    系统能从聊天频次和时间间隔里发现这种变化，主动提醒你维护那些你不想丢掉的人。

    4. 跨对话信息回溯¶

    微信原生搜索在这种场景下基本不够用。
    Agent 可以跨所有对话做语义回溯，直接帮你找出“谁在什么时候说过什么”。

    5. 情绪感知与交互自适应¶

    如果你今天明显很累、回消息很慢、文本很短、几乎没有表情，那系统不该再机械催你做事。
    它应该切换到另一套交互策略。

    6. 生活轨迹时间轴¶

    很多人的生活记录做不起来，不是因为不想记，而是因为太麻烦。
    如果聊天本身就能反推出你的行动轨迹，那么时间轴和生活日志就可以自动生成。

    7. 自动日记系统¶

    晚上自动总结当天发生了什么、见了谁、情绪有什么波动、哪些承诺还没履行。
    这件事一旦跑通，长期价值会非常高。

    8. IoT 联动控制¶

    提醒你起床只是初级形态。
    真正的联动是：提醒无效后，Agent 直接去控制窗帘、音乐、灯光等物理环境。

    9. 社交日程整合¶

    从聊天中抽取约饭、开会、聚会和临时承诺，自动做冲突检测和提前提醒。
    这比你自己手抄到日历里更接近真实生活。

    为什么这条路线比普通 bot 方案高一个量级¶

    很多现成方案已经能做到“接微信、发微信、跑模型”。
    但真正的差距，不在能不能收发消息，而在数据视野。

    能力维度 普通 bot 路线 这套 WeFlow + ilink + Agent 路线
    数据来源 仅 bot 对话 全部微信聊天
    上下文理解 只知道你告诉它的事 知道你的整个社交环境
    时间追踪 依赖手动汇报 从聊天行为自动推断
    打断策略 随机轮询 基于实时上下文判断
    承诺追踪 做不到 可以自动提取
    关系管理 做不到 可以建立长期图谱
    信息检索 只能搜 bot 对话 能跨所有聊天搜索
    情绪判断 仅看你发给 bot 的文本 可结合更完整的行为元数据
    输出能力 会发消息 会发消息，也能扩展到报表、前端、IoT

    这就是为什么我认为这不是“又一个微信自动化项目”，而是一条明显更有想象力的个人 Agent 路线。

    这件事最现实的边界¶

    这套东西很强，但边界也必须说清楚。

    低风险部分¶

    • ilink 收发消息
    • 本地 Agent 运行
    • 本地前端和本地存储

    这些部分本质上都还是个人工作流增强。

    中风险部分¶

    • WeFlow 读取本地微信数据
    • 第三方接口和框架的稳定性

    这类风险主要是技术灰度和平台策略变化风险。

    高风险部分¶

    • 把这套系统做成公开产品
    • 处理他人聊天数据并对外提供服务
    • 商业化部署

    一旦进入产品化和商业化，就不再只是技术问题，而是隐私、合规、平台协议和法律责任问题。

    所以我现在的判断很明确：

    自己用，狠狠干；对外做产品，先别在微信上硬刚。

    更稳妥的路线是先在自己场景里把逻辑跑通，把能力验证清楚，再考虑把相同的 Agent 决策层迁移到 Telegram、自建 IM 或别的可控环境。

    现在最值得继续补的，不是“再换一个模型”¶

    很多人做到这一步会开始纠结：

    • Claude 还是 Codex？
    • OpenClaw 还是别的 Agent 框架？
    • 要不要多 Agent？

    这些都重要，但不是当前阶段的第一优先级。

    我认为接下来最值钱的三件事是：

    1. 把承诺提取做成稳定模块
    2. 把关系衰减和时间轴真正落盘
    3. 把输出层做成一个能长期看的本地前端

    因为模型随时可以换，真正难替代的是：

    • 你自己的数据闭环
    • 你的行为结构
    • 你的长期记忆资产

    谁先把这三件事做出来，谁就不是在玩 demo，而是在做个人 Agent 基建。

    最后¶

    这个项目让我最兴奋的地方，不是“AI 接了微信”，而是它开始逼近一个更真实的方向：

    Agent 不再只理解你说给它听的话，
    而是开始理解你的人际关系、时间分配、承诺系统和生活惯性。

    这才像一个真正的个人 AI 助手。

    不是聊天玩具。
    不是自动回复脚本。
    而是一套有机会长期陪着你、管理你、提醒你、甚至干预你生活节奏的系统。

    这条路线，我会继续往下打。
